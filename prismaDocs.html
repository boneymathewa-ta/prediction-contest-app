Prisma Schema Design

A hands-on guide for your Sports Prediction Platform — Tournament, Match, User and beyond. Built for developers using Cursor AI.

1. Think Before You Type

The biggest mistake developers make is opening VS Code before thinking. Prisma schema errors are cheap to fix. Logic errors discovered on Day 3 are not.

Answer these questions in plain English first:

🏆 Tournament

Has a name, sport type, start/end dates.
Contains many Matches.
Can be active or completed.

⚽ Match

Belongs to one Tournament.
Has two teams, a date/time.
Has one Contest.
Has a result (or not yet).

🎯 Contest

Belongs to one Match.
Has many Questions.
Has an expiry time.
Can be open or closed.

👤 User

Logs in via SSO (Google).
Submits Predictions.
Earns points.
Appears on Leaderboard.

Tournament ──has many──▶ Match
                                └──has one──▶ Contest
                                                 └──has many──▶ Question
                                                                 └──has many──▶ Prediction ◀──belongs to── User
⚠️ Key Design Decision

A Prediction belongs to both a Question and a User. One user can only submit one prediction per question (upsert on change). This is your most critical constraint — get it wrong and your scoring breaks.

3. Schema File Basics

Every schema.prisma starts with two blocks before any models:

// prisma/schema.prisma

// 1. Tells Prisma which database to use
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

// 2. Tells Prisma how to generate its client code
generator client {
  provider = "prisma-client-js"
}

After these two blocks, you define your models. Each model = one database table.

Anatomy of a Model Field

model Example {
  //  fieldName    DataType   Attributes
      id          String     @id @default(cuid())
      name        String

}

4. Tournament Model — Step by Step

Let's build this model one concept at a time, not all at once.

Start with just the fields

model Tournament {
  id        String    @id @default(cuid())
  name      String    // "IPL 2026", "Cricket World Cup 2026"
  sport     Sport     // enum: CRICKET or FOOTBALL
  startDate DateTime
  endDate   DateTime
  logoUrl   String?   // optional — might not have logo yet
  status    TournamentStatus @default(UPCOMING)
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  // Relations (we'll explain these below)
  matches   Match[]
}
💡 Why cuid() not autoincrement()?

cuid() generates unique string IDs like clx7abc123. They're safe to expose in URLs, harder to guess than sequential integers (1, 2, 3...), and work across distributed systems. Use cuid() for all your IDs.

ℹ️ @updatedAt

This is magic — Prisma automatically updates this field every time you update the record. You never set it manually. Always add it to every model.

5. Match Model — Adding Relations

Match is where relations get interesting. It belongs to a Tournament and has a Contest.

model Match {
  id           String       @id @default(cuid())
  homeTeam     String       // "Chennai Super Kings"
  awayTeam     String       // "Mumbai Indians"
  matchDate    DateTime
  venue        String?
  round        String?      // "Group Stage", "Semi Final" etc
  status       MatchStatus  @default(UPCOMING)

  // Result fields — null until match is completed
  homeScore    String?      // "185/4" for cricket, "2" for football
  awayScore    String?
  result       String?      // "CSK won by 23 runs"

  // ── Relation to Tournament ──────────────────────
  // Step 1: Foreign key column (stored in DB)
  tournamentId String
  // Step 2: Relation field (not in DB, just for Prisma queries)
  tournament   Tournament   @relation(fields: [tournamentId], references: [id])

  // ── Relation to Contest ─────────────────────────
  // One match has one contest (optional — may not have one yet)
  contest      Contest?

  createdAt    DateTime     @default(now())
  updatedAt    DateTime     @updatedAt
}

Understanding the @relation Syntax

This is the part that confuses most beginners. Let's break it apart:

// In Match model:
tournamentId String                          ← Real DB column (foreign key)
tournament   Tournament  @relation(            ← Virtual (only for Prisma)
               fields: [tournamentId],       ← "use THIS column..."
               references: [id]            ← "...to find THIS field in Tournament"
             )

// In Tournament model (the other side):
matches  Match[]   ← "I have many matches" — no @relation needed on this side
⚠️ The Golden Rule of Relations

The child model (Match) stores the foreign key (tournamentId) and has the @relation attribute. The parent model (Tournament) just declares the array (matches Match[]). Never the other way around.

6. User Model — Auth-Aware Design

Since you're using SSO (Clerk or NextAuth), your User model needs to link to the external auth provider.

model User {
  id          String    @id @default(cuid())

  // If using Clerk: store Clerk's userId here
  // If using NextAuth: this is auto-handled
  clerkId     String?   @unique   // Clerk's user ID

  email       String    @unique
  name        String?
  avatarUrl   String?

  // Stats (updated after each contest is scored)
  totalPoints      Int  @default(0)
  currentStreak    Int  @default(0)
  totalPredictions Int  @default(0)
  correctPredictions Int @default(0)

  // Relations
  predictions Prediction[]

  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  @@index([totalPoints])  // Index for fast leaderboard queries
}
ℹ️ Storing Stats on User vs Calculating on the Fly

For a 3-day sprint, store totalPoints directly on User. Update it when you score a contest. Don't try to calculate it from all predictions on every leaderboard load — that's slow and complex. Denormalize for speed.

8. Full Schema — All Models

✅ Copy this into your prisma/schema.prisma

This is the complete schema for your MVP. Every field has been chosen deliberately. Read the comments — they explain the decisions.

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

// ─────────────────────────────────────────
//  ENUMS
// ─────────────────────────────────────────

enum Sport {
  CRICKET
  FOOTBALL
}

enum TournamentStatus {
  UPCOMING
  ACTIVE
  COMPLETED
}

enum MatchStatus {
  UPCOMING
  LIVE
  COMPLETED
  ABANDONED
}

enum ContestStatus {
  OPEN        // accepting predictions
  LOCKED      // expired, no more submissions
  SCORED      // results entered, points awarded
}

enum ContestType {
  DAILY       // per-match contest
  MEGA        // tournament-level contest
}

enum QuestionType {
  MATCH_WINNER
  PLAYER_OF_MATCH
  FIRST_WICKET
  FIRST_GOAL_SCORER
  TOP_BATSMAN
  TOP_BOWLER
  EXACT_SCORE
  CUSTOM       // admin-defined free-form question
}

// ─────────────────────────────────────────
//  TOURNAMENT
// ─────────────────────────────────────────

model Tournament {
  id          String           @id @default(cuid())
  name        String           // "IPL 2026"
  sport       Sport
  startDate   DateTime
  endDate     DateTime
  logoUrl     String?
  status      TournamentStatus @default(UPCOMING)
  hasGroups   Boolean          @default(false) // false for IPL, true for WC

  matches     Match[]
  contests    Contest[]        // includes mega contests

  createdAt   DateTime         @default(now())
  updatedAt   DateTime         @updatedAt
}

// ─────────────────────────────────────────
//  MATCH
// ─────────────────────────────────────────

model Match {
  id            String      @id @default(cuid())
  homeTeam      String
  awayTeam      String
  matchDate     DateTime    // always UTC
  venue         String?
  round         String?     // "Group Stage", "Semi Final", "Final"
  groupName     String?     // "Group A" — null for IPL
  status        MatchStatus @default(UPCOMING)

  // Result — null until match is completed
  homeScore     String?     // "185/4 (20 overs)" or "2"
  awayScore     String?
  resultSummary String?     // "CSK won by 23 runs"
  winnerId      String?     // team name of winner (for scoring)

  // Relations
  tournamentId  String
  tournament    Tournament  @relation(fields: [tournamentId], references: [id])
  contest       Contest?

  createdAt     DateTime    @default(now())
  updatedAt     DateTime    @updatedAt

  @@index([tournamentId, matchDate])
}

// ─────────────────────────────────────────
//  CONTEST
// ─────────────────────────────────────────

model Contest {
  id           String       @id @default(cuid())
  title        String       // "CSK vs MI — Match Predictions"
  type         ContestType  @default(DAILY)
  status       ContestStatus @default(OPEN)
  expiresAt    DateTime     // after this → status = LOCKED

  // DAILY contest links to a match
  // MEGA contest links only to tournament
  matchId      String?      @unique  // @unique = one contest per match
  match        Match?       @relation(fields: [matchId], references: [id])

  tournamentId String
  tournament   Tournament   @relation(fields: [tournamentId], references: [id])

  questions    Question[]

  createdAt    DateTime     @default(now())
  updatedAt    DateTime     @updatedAt
}

// ─────────────────────────────────────────
//  QUESTION
// ─────────────────────────────────────────

model Question {
  id            String       @id @default(cuid())
  text          String       // "Who will win the match?"
  type          QuestionType @default(CUSTOM)
  options       String[]     // ["CSK", "MI", "Draw"] — Postgres array
  correctAnswer String?      // set after match. null until then
  points        Int          @default(10)   // base points for correct answer
  bonusPoints   Int          @default(0)    // extra for exact/rare predictions
  order         Int          @default(0)    // display order within contest

  contestId     String
  contest       Contest      @relation(fields: [contestId], references: [id])

  predictions   Prediction[]

  createdAt     DateTime     @default(now())
}

// ─────────────────────────────────────────
//  PREDICTION  (the heart of the app)
// ─────────────────────────────────────────

model Prediction {
  id          String    @id @default(cuid())
  answer      String    // User's chosen option
  isCorrect   Boolean?  // null until contest is scored
  pointsEarned Int     @default(0) // set when scored

  userId      String
  user        User      @relation(fields: [userId], references: [id])

  questionId  String
  question    Question  @relation(fields: [questionId], references: [id])

  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  // ← THIS IS CRITICAL
  // Ensures one prediction per user per question
  // Enables upsert (update if exists, create if not)
  @@unique([userId, questionId])
  @@index([userId])
  @@index([questionId])
}

// ─────────────────────────────────────────
//  USER
// ─────────────────────────────────────────

model User {
  id                 String       @id @default(cuid())
  clerkId            String?      @unique  // from Clerk SSO
  email              String       @unique
  name               String?
  avatarUrl          String?
  totalPoints        Int          @default(0)
  currentStreak      Int          @default(0)
  totalPredictions   Int          @default(0)
  correctPredictions Int          @default(0)

  predictions        Prediction[]

  createdAt          DateTime     @default(now())
  updatedAt          DateTime     @updatedAt

  @@index([totalPoints])  // fast leaderboard sorting
}

9. Enums — Why They Matter

Enums prevent bad data from ever entering your database. Without them, someone can set status = "compleeted" (typo) and your app silently breaks.

❌ Without Enum

status String
// Could be:
// "active", "Active", "ACTIVE"
// "compleeted" (typo)
// "done", "finished"
// anything!

✅ With Enum

enum Status {
  OPEN
  LOCKED
  SCORED
}
// DB rejects anything else.
// TypeScript autocompletes it.
// No bugs possible.

10. Running Migrations

Migrations are how your schema becomes actual database tables. Prisma tracks every migration in a prisma/migrations/ folder.

1 First migration (development)
npx prisma migrate dev --name init

This creates the migration file AND runs it immediately. Use during development.

2 After changing schema, create a new migration
# Example: you added a new field to Contest
npx prisma migrate dev --name add_contest_banner_url
3 Generate Prisma Client (after schema changes)
npx prisma generate

This regenerates TypeScript types. Cursor will autocomplete your models perfectly after this.

4 View your database in a browser
npx prisma studio

Opens a visual database explorer at localhost:5555. Extremely useful during development to verify your data.

🚫 Never Edit Migration Files Manually

Prisma tracks migration history. If you edit a migration file that's already been run, Prisma gets confused and breaks. Instead, always create a new migration for each schema change.

11. The Queries You'll Actually Use

Create a src/lib/prisma.ts singleton first — never create a new PrismaClient in every file:

// src/lib/prisma.ts
import { PrismaClient } from '@prisma/client'

const globalForPrisma = global as { prisma?: PrismaClient }

export const prisma =
  globalForPrisma.prisma ?? new PrismaClient({ log: ['query'] })

if (process.env.NODE_ENV !== 'production')
  globalForPrisma.prisma = prisma

Query 1: Get matches for a tournament (Schedules page)

const matches = await prisma.match.findMany({
  where: {
    tournamentId: 'tournament-id-here',
    status: { in: ['UPCOMING', 'LIVE'] }
  },
  include: {
    contest: {                    // also fetch the contest
      select: { id: true, status: true, expiresAt: true }
    }
  },
  orderBy: { matchDate: 'asc' }
})

Query 2: Get contest with questions (Contest page)

const contest = await prisma.contest.findUnique({
  where: { id: contestId },
  include: {
    questions: {
      orderBy: { order: 'asc' }
    },
    match: {
      select: { homeTeam: true, awayTeam: true, matchDate: true }
    }
  }
})

Query 3: Submit / Update a prediction (upsert)

// This is your most important query.
// Creates the prediction if first time.
// Updates it if user changes their answer.
// ONLY works because of @@unique([userId, questionId])

const prediction = await prisma.prediction.upsert({
  where: {
    userId_questionId: { userId, questionId }   // compound unique
  },
  update: { answer: newAnswer },
  create: { userId, questionId, answer: newAnswer }
})

Query 4: Leaderboard (sorted by points)

const leaderboard = await prisma.user.findMany({
  select: {
    id: true, name: true, avatarUrl: true,
    totalPoints: true, currentStreak: true,
    correctPredictions: true, totalPredictions: true
  },
  orderBy: { totalPoints: 'desc' },
  take: 50   // top 50 only
})

Query 5: Score a contest (after match result entered)

// Run this after admin sets correctAnswer on all questions
async function scoreContest(contestId: string) {
  const questions = await prisma.question.findMany({
    where: { contestId },
    include: { predictions: true }
  })

  for (const question of questions) {
    for (const prediction of question.predictions) {
      const isCorrect = prediction.answer === question.correctAnswer
      const pointsEarned = isCorrect ? question.points : 0

      // Update prediction
      await prisma.prediction.update({
        where: { id: prediction.id },
        data: { isCorrect, pointsEarned }
      })

      // Update user total (increment, not recalculate)
      if (isCorrect) {
        await prisma.user.update({
          where: { id: prediction.userId },
          data: {
            totalPoints: { increment: pointsEarned },
            correctPredictions: { increment: 1 }
          }
        })
      }
    }
  }

  // Mark contest as scored
  await prisma.contest.update({
    where: { id: contestId },
    data: { status: 'SCORED' }
  })
}

12. Seed Script

A seed script populates your DB with test data so you can build UI without waiting for real data. Create this at prisma/seed.ts:

// prisma/seed.ts
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()

async function main() {
  // 1. Create tournament
  const tournament = await prisma.tournament.create({
    data: {
      name: 'IPL 2026',
      sport: 'CRICKET',
      startDate: new Date('2026-03-22'),
      endDate: new Date('2026-05-26'),
      status: 'ACTIVE',
      hasGroups: false
    }
  })

  // 2. Create a match with a contest and questions (nested create)
  const match = await prisma.match.create({
    data: {
      homeTeam: 'Chennai Super Kings',
      awayTeam: 'Mumbai Indians',
      matchDate: new Date('2026-03-25T14:00:00Z'),
      venue: 'Wankhede Stadium',
      tournamentId: tournament.id,

      // Nested: create contest + questions in one query!
      contest: {
        create: {
          title: 'CSK vs MI — Match Predictions',
          type: 'DAILY',
          expiresAt: new Date('2026-03-25T13:30:00Z'),
          tournamentId: tournament.id,
          questions: {
            create: [
              {
                text: 'Who will win the match?',
                type: 'MATCH_WINNER',
                options: ['Chennai Super Kings', 'Mumbai Indians'],
                points: 10,
                order: 1
              },
              {
                text: 'Who will be Player of the Match?',
                type: 'PLAYER_OF_MATCH',
                options: ['MS Dhoni', 'Rohit Sharma', 'Ruturaj Gaikwad', 'Hardik Pandya'],
                points: 20,
                order: 2
              },
              {
                text: 'Who will take the first wicket?',
                type: 'FIRST_WICKET',
                options: ['Jasprit Bumrah', 'Deepak Chahar', 'Mustafizur Rahman'],
                points: 30,
                order: 3
              }
            ]
          }
        }
      }
    }
  })

  // 3. Create test users
  const user1 = await prisma.user.create({
    data: { email: 'test@example.com', name: 'Test User' }
  })

  console.log('✅ Seeded:', { tournament: tournament.name, match: match.id })
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect())

Add to package.json:

"prisma": {
  "seed": "ts-node prisma/seed.ts"
}

Run with: npx prisma db seed

13. Common Mistakes (Save Yourself Hours)

MistakeWhat Goes WrongFix
Multiple new PrismaClient() Hits connection limit (Postgres max ~100 connections). App crashes under load. Always use the singleton from lib/prisma.ts
Missing @@unique on Prediction Users can submit duplicate predictions. Scoring gives double points. Always have @@unique([userId, questionId])
Fetching all relations always Leaderboard query fetches 10,000 predictions for every page load. Use select to only get fields you need
Checking contest expiry on frontend only User changes system clock, bypasses deadline. Always check contest.expiresAt < new Date() in the API route
Storing local timezone dates Match shows wrong time for users in different timezones. Always store UTC (new Date().toISOString()). Convert to local on display.
Editing migration files Prisma migration history breaks. Schema and DB fall out of sync. Never edit existing migrations. Create new ones.

14. Using This Schema with Cursor

Once you have your schema.prisma and have run npx prisma generate, Cursor becomes dramatically more powerful because it can see your exact types.

Best Cursor Prompts for Database Work

Prompt template for API routes
Using the Prisma schema in prisma/schema.prisma and the singleton 
from src/lib/prisma.ts, create a Next.js App Router API route at 
app/api/contests/[contestId]/predict/route.ts that:

1. Accepts POST with body { questionId, answer }
2. Gets the current user from Clerk session
3. Checks the contest.expiresAt is in the future
4. Upserts the prediction using @@unique([userId, questionId])
5. Returns the saved prediction
6. Returns 400 if contest is expired, 401 if not logged in
Prompt template for React components
Using the Prisma types from @prisma/client, create a TypeScript 
React component ContestCard.tsx that:

- Accepts props: contest: Contest & { questions: Question[], match: Match }
- Shows match teams, date, and countdown timer to expiresAt
- Lists each question with radio button options
- Submit button calls POST /api/contests/{id}/predict
- Disables form if contest.status !== 'OPEN'
- Uses Tailwind CSS for styling
✅ Pro Tip

After running npx prisma generate, import types directly from Prisma in your prompts: import type { Contest, Question, User } from '@prisma/client'. Cursor sees these types and generates code that perfectly matches your schema — no type mismatches.
